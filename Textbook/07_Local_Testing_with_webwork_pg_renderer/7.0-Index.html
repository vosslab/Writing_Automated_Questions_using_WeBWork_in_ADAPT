This chapter is about a faster way to preview and debug WeBWorK problems while you are writing them. It is most useful when ADAPT preview fails with a generic error and you need a line number and a real message.

This chapter introduces webwork-pg-renderer as a lightweight way to preview and test problems locally. The goal is not to replace full testing in WeBWorK or ADAPT, but to give you a tight edit-preview loop for day-to-day authoring.

The renderer uses the same PG 2.17 subset as ADAPT, so missing macros here are missing in ADAPT as well. Use local rendering to catch subset issues before you import into ADAPT.

What this tool is (and is not)

  • It is: a local renderer for PG and PGML that helps you iterate quickly with reproducible seeds.
  • It is not: a full WeBWorK deployment, course manager, or replacement for ADAPT assignment preview.

When to use local rendering

If you need to... Use... Because...
Fix a render failure with a real error message Local renderer You get line-level errors and logs.
Sweep random variants quickly Local renderer You can change seeds fast and keep failures reproducible.
Confirm assignment behavior and student permissions ADAPT preview Only ADAPT shows the final course context.

Minimal prerequisites

  • Podman or Docker installed on your machine.
  • A local folder that contains your .pg files.
  • A quick health check: GET /health should return a 200 status.

The one rule that saves hours

When a problem fails, write down (1) the file path and (2) the seed. Do not change either one until the failure is fixed.

Apply it today

  • Use local rendering first for line-level errors and rapid iteration.
  • Use ADAPT preview last to confirm assignment context and student view.
