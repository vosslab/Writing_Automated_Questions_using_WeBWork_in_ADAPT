Local rendering works best when it feels like editing a document: open a problem file, render it, make a small change, and render again. This section outlines a simple workflow you can repeat every day.

Think of this like a lab workflow. You keep one sample (a single .pg file), one condition (a fixed seed), and you change one variable at a time. The payoff is that you can reproduce the same failure and fix it without guessing.

First run checklist (5 minutes)

  • Check the renderer status with GET /health.
  • Open the editor page and load private/yourfile.pg.
  • Set a seed (start with something memorable like 1234).
  • Render once before editing so you know what "working" looks like.
  • Make one small text edit, save, then render again.

Choose one file and make it reproducible

  • Pick one .pg file to work on and keep the path stable while you iterate.
  • Use a fixed seed when you are debugging so the same random variant renders each time.
  • Change one thing per iteration so you can tell what caused the difference.

Private folder workflow

Place your files inside the private/ folder that is mounted into the renderer. The web editor at http://localhost:3000 can then load the file path directly.

What to record while you work

Record Example Why it matters
File path private/enzyme_kinetics.pg Keeps your failing case tied to one file.
Seed 1234 Locks the random variant so you can reproduce the same render.
What you changed "Changed units from mM to μM in the prompt" Makes debugging traceable when grading stops matching the story.

Render and check the student-facing prompt

  • Render the problem and read it like a student would read it.
  • Confirm the story makes sense with the chosen randomized values (units, ranges, and rounding).
  • Confirm every answer blank is visible, in the right order, and has the right unit/format cues nearby.

What "done" looks like for one edit

  • The prompt reads cleanly as a student-facing question.
  • The values are plausible for biology (units, ranges, rounding).
  • Every blank appears where expected and matches the intended answer.

Save edits back to disk

  • Keep edits small and frequent rather than making one large rewrite.
  • When you find a failing case, keep the file path and seed so you can reproduce it later.
  • Once the local render looks right, move to course-level testing for the final check.

Apply it today

  • Use local rendering for the tight edit-preview loop, then use ADAPT for the final publish workflow.
  • When debugging, write down the failing seed and do not change it until the failure is fixed.
